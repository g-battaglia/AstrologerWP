# F0.5 — Spike Interactivity API (validazione early)

> **Theme:** Prima di investire settimane in Interactivity API per F6, validare con uno spike che copra i casi più complessi (form multi-step + SVG injection + state machine).
> **Effort:** XS (0.5 giornata)
> **Dipendenze:** F0

---

## Obiettivo

Un **blocchetto throwaway** chiamato `astrologer-api/spike-birth-form` che:
1. Renderizza un form con 4 campi (name, date, lat, lng).
2. Usa uno store Interactivity per gestire state machine (`idle → validating → submitting → success | error`).
3. Al submit fa POST a un endpoint mock `wp-json/astrologer/v1/spike` (anch'esso fake in questa fase).
4. Riceve un SVG inline mock e lo inietta nel DOM tramite `data-wp-bind--inner-html` con sanitizzazione.
5. Mostra un fallback errore.

Obiettivo **non** è implementare la feature — è **validare l'architettura**. Il codice dello spike sarà cancellato o spostato in `tests/spikes/` a fine fase.

### F0.5.3 — Interactivity store

**File:** `blocks/spike-birth-form/view.ts`

```ts
import { store, getContext } from '@wordpress/interactivity';

const { state, actions } = store('astrologer/spike', {
  state: {
    status: 'idle',
    fields: { name: '', date: '', lat: '', lng: '' },
    svg: '',
    error: '',
  },
  actions: {
    *submit(e: SubmitEvent) {
      e.preventDefault();
      state.status = 'submitting';
      try {
        const res = yield fetch('/wp-json/astrologer/v1/spike', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json', 'X-WP-Nonce': (window as any).wpApiSettings?.nonce },
          body: JSON.stringify(state.fields),
        });
        const data = yield res.json();
        state.svg = data.svg;
        state.status = 'success';
      } catch (err: any) {
        state.error = err.message;
        state.status = 'error';
      }
    },
  },
});
```

Note:
- Generator syntax (`*submit`) è il pattern async Interactivity API.
- `yield` su Promise.
- `state` mutazioni trigger re-render diretto.

## Criterio di successo (checkpoint #2)

✅ **OK**: tutti i 5 punti manuali passano, E2E verde. Procedi a F1.

❌ **NOK**: problemi insuperabili con Interactivity API (es. state machine troppo verbosa per 7 form complessi, SVG injection XSS-safe impossibile, DX sotto soglia tollerabile).
**Decisione fallback**: mantenere React anche in frontend per i form, usare Interactivity solo per display statici leggeri. Questo riduce F6 effort di ~40% ma aumenta bundle frontend di ~100KB.

In entrambi i casi, **documenta l'esito nel commit message** di chiusura F0.5.

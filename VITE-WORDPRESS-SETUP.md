# Vite + React integration in WordPress (Astrologer WP)

This document explains **step by step** how Vite (dev & build) was integrated
into the `astrologer-wp` WordPress plugin, and how to reproduce / reuse this
configuration in the future.

The goal is to:

- use **Vite** to develop a modern React/TSX frontend (with Tailwind/shadcn),
- mount React components inside WordPress via **shortcodes** and **Gutenberg blocks**,
- have two modes:
  - **dev** (Vite dev server on `http://localhost:5173`),
  - **build** (static bundle in `frontend/dist/assets/*` loaded via `wp_enqueue_script/style`).

---

## 1. Relevant file structure

Inside the plugin:

- `frontend/`
  - `src/main.tsx` → React entry that mounts components into WP containers
  - `src/ComponentMounter.tsx` → selects which React component to render (BirthForm, etc.)
  - `src/lib/utils.ts` → helpers `apiRequest`, `getConfig`, `t`
  - `vite.config.ts` → Vite config for build/dev
  - `package.json` → `dev`/`build` scripts
  - `dist/` → build output (`assets/index.js`, `assets/main.css`, manifest)
- `includes/class-astrologer-api-frontend.php` → handles asset loading
- `includes/class-astrologer-api-blocks.php` → shortcodes + blocks that generate `data-astrologer-component` containers

## 2. Vite: general configuration (`frontend/vite.config.ts`)

```ts
import { defineConfig } from 'vite'
import path from 'path'

/**
 * Vite configuration for the WordPress frontend.
 *
 * The build produces a JS + CSS bundle in dist/assets/ which is
 * loaded by the PHP plugin via wp_enqueue_script/style.
 *
 * Note: we DON'T use @vitejs/plugin-react in this setup because its
 * React Refresh preamble needs control over the input HTML, which
 * WordPress manages independently. Vite/esbuild still handles TSX/JSX
 * just fine.
 */
export default defineConfig({
  // Aliases for clean imports
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },

  // Build config for WordPress
  build: {
    outDir: 'dist',
    manifest: true,
    rollupOptions: {
      input: 'src/main.tsx',
      output: {
        entryFileNames: 'assets/index.js',
        chunkFileNames: 'assets/[name].js',
        assetFileNames: 'assets/[name][extname]',
      },
    },
    minify: 'terser',
    sourcemap: false,
  },

  // Dev server
  server: {
    port: 5173,
    strictPort: true,
    cors: true,
  },
})
```

### Key points

- The JS entry is **fixed**: `assets/index.js`.
- The generated CSS ends up in `assets/main.css` (thanks to the `assetFileNames` naming strategy).
- **No** `@vitejs/plugin-react` → no React Refresh preamble, compatible with HTML generated by WordPress.

## 3. React entry: `src/main.tsx`

```tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import { ComponentMounter } from './ComponentMounter'
import './index.css'

function initAstrologerComponents(): void {
  const containers = document.querySelectorAll<HTMLElement>('[data-astrologer-component]')

  if (containers.length === 0) {
    console.log('[Astrologer API] No components found on the page.')
    return
  }

  console.log(`[Astrologer API] Found ${containers.length} components to mount.`)

  containers.forEach((container) => {
    const componentType = container.dataset.astrologerComponent
    const propsJson = container.dataset.props || '{}'

    if (!componentType) {
      console.warn('[Astrologer API] Container without component type:', container)
      return
    }

    try {
      const props = JSON.parse(propsJson)

      const root = createRoot(container)
      root.render(
        <StrictMode>
          <ComponentMounter type={componentType} props={props} />
        </StrictMode>
      )

      console.log(`[Astrologer API] Mounted component: ${componentType}`)
    } catch (error) {
      console.error(
        `[Astrologer API] Error parsing props for ${componentType}:`,
        error
      )
    }
  })
}

if (document.readyState === 'loading') {
  document.addEventListener('DOMContentLoaded', initAstrologerComponents)
} else {
  initAstrologerComponents()
}
```

## 4. PHP bridge for assets: `class-astrologer-api-frontend.php`

### 4.1. Asset registration

```php
public function register_assets(): void {
    $dist_path = ASTROLOGER_API_PLUGIN_DIR . 'frontend/dist/';
    $dist_url  = ASTROLOGER_API_PLUGIN_URL . 'frontend/dist/';

    // Check if the production bundle exists.
    // If it does NOT exist and WP_DEBUG is active, use dev mode (Vite).
    if ( ! file_exists( $dist_path . 'assets/index.js' ) ) {
        if ( defined( 'WP_DEBUG' ) && WP_DEBUG ) {
            $this->register_dev_assets();
        }
        return;
    }

    // Register CSS (generated by Vite)
    // Vite may generate "main.css" or "index.css" depending on config.
    $css_handle = 'astrologer-api-frontend';
    if ( file_exists( $dist_path . 'assets/main.css' ) ) {
        wp_register_style(
            $css_handle,
            $dist_url . 'assets/main.css',
            array(),
            ASTROLOGER_API_VERSION
        );
    } elseif ( file_exists( $dist_path . 'assets/index.css' ) ) {
        wp_register_style(
            $css_handle,
            $dist_url . 'assets/index.css',
            array(),
            ASTROLOGER_API_VERSION
        );
    }

    wp_register_script(
        'astrologer-api-frontend',
        $dist_url . 'assets/index.js',
        array(),
        ASTROLOGER_API_VERSION,
        true
    );

    add_filter( 'script_loader_tag', array( $this, 'add_module_type' ), 10, 3 );

    wp_localize_script(
        'astrologer-api-frontend',
        'astrologerApiConfig',
        $this->get_frontend_config()
    );
}
```

### 4.2. Dev assets (Vite)

```php
private function register_dev_assets(): void {
    $vite_server = 'http://localhost:5173';

    wp_register_script(
        'vite-client',
        $vite_server . '/@vite/client',
        array(),
        null,
        false
    );

    wp_register_script(
        'astrologer-api-frontend',
        $vite_server . '/src/main.tsx',
        array( 'vite-client' ),
        null,
        true
    );

    add_filter( 'script_loader_tag', array( $this, 'add_module_type' ), 10, 3 );

    wp_localize_script(
        'astrologer-api-frontend',
        'astrologerApiConfig',
        $this->get_frontend_config()
    );
}
```

### 4.3. Enqueue

```php
public static function enqueue_frontend_assets(): void {
    if ( self::$assets_enqueued ) {
        return;
    }

    if ( ! wp_script_is( 'astrologer-api-frontend', 'registered' ) ) {
        ( new self() )->register_assets();
    }

    if ( wp_style_is( 'astrologer-api-frontend', 'registered' ) ) {
        wp_enqueue_style( 'astrologer-api-frontend' );
    }

    if ( wp_script_is( 'astrologer-api-frontend', 'registered' ) ) {
        wp_enqueue_script( 'astrologer-api-frontend' );
    }

    self::$assets_enqueued = true;
}
```

## 5. How to use dev and build in the future

### Development mode (dev, with Vite)

1. In `wp-config.php` enable debug:

   ```php
   define( 'WP_DEBUG', true );
   ```

2. Make sure the build **does not** exist:

   ```bash
   cd frontend
   rm -rf dist
   ```

3. Start the Vite dev server:

   ```bash
   cd frontend
   npm run dev   # or bun run dev
   ```

4. Open a WordPress page with an Astrologer shortcode/block.
   - The page will load:
     - `http://localhost:5173/@vite/client`
     - `http://localhost:5173/src/main.tsx`
   - In the console you’ll see messages like:
     - `[Astrologer API] Found X components to mount.`
     - `[Astrologer API] Mounted component: birth-form`

5. **Autorefresh / HMR**:
   - With the Vite dev server running and the WordPress page open,
     every time you edit and save a file under `frontend/src/**`:
     - Vite sends an update via WebSocket to the browser.
     - If possible, it applies HMR only to the updated module.
     - Otherwise it performs an automatic `location.reload()`.
   - In practice: **you save the React/TSX file and the WordPress page updates by itself**,
     without needing a manual refresh.

### Production mode (build)

1. From the frontend:

   ```bash
   cd frontend
   npm run build   # or bun run build
   ```

   This generates:

   - `frontend/dist/assets/index.js`
   - `frontend/dist/assets/main.css`

2. Even if `WP_DEBUG` is true, the presence of `dist/assets/index.js` makes the plugin
   **always** use the production build.

3. Reload the WordPress page (clear browser cache) and check that assets are served from
   `/wp-content/plugins/.../frontend/dist/assets/...`.

## 6. Steps to reuse in another plugin

1. Create a `frontend/` folder with:
   - `src/main.tsx` structured like above.
   - `vite.config.ts` that emits `assets/index.js` and `assets/main.css`.
   - `package.json` with `dev` and `build` scripts.

2. Create a PHP class similar to `Astrologer_API_Frontend` that:
   - registers build or dev in `register_assets()`;
   - has an `enqueue_frontend_assets()` method to call in all shortcode/block renders.

3. In shortcode/block render functions, output `<div data-...>` containers and call the enqueue.

4. Use `wp_localize_script` to pass configuration and i18n to the frontend.

With these steps you have a reusable guide to integrate Vite+React into any WordPress plugin,
with the same ergonomics you currently have in `astrologer-wp`.
